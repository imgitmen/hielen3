    • Introduzione della classe marshmallow "ColorMap" per la serializzazione delle informazioni custom di colorazione dei dati: tra i vari campi che possono essere ritornati da actionschemata ora ne esiste uno chiamato ColorMap. Es.: PhotoMonitoring actionschemata

    Come funziona ColorMap:

    Incontrando il campo definito come "ColorMap" all'interno di una risposta di actionschemata, il front-end aprir l'interfaccia di scelta dei colori. Il risultato dell'interazione dovr essere sintetizzato in un json da fronire come parametro all'azione corrispondente. Il json dovr essere nella forma di un array di lunghezza variabile di COPPIE (valore,colore):

    [
    [
        valore,
        colore
    ],
    [
        valore,
        colore,
    ]
    ...
]

    dove:

    valore un numero di qualsiasi genere con punto decimale

    colore una stringa di colore hex

    esempio:

    [
    [
        -0.25,
        '#FFFF00'
    ],
    [
        -0.125,
        '#00FF00'
    ],
    [
        0,
        '#0000FF'
    ],
    [
        0.5,
        '#FF00FF'
    ],
    [
        1,
        '#AEBD00'
    ],
                                            ]

    Un array del genere verrebbe inteso in questo modo:

    [-inf,-0.25) :

    [-0.25,-0.125) : -->

    [-0.125,0.0) : -->

    [0.0,0.5) : -->

    [0.5,1.0) : -->

    [1.0,inf) :

    L'informazione sulla colormap pu essere associata ad ongi singola serire dati e pu essere fornita in front-end su richiesta: allo scopo L'API features stata modificata in modo da fornire la colormap insieme con i parametri delle features specificando l'opzione info=parameters,...

    esempio (vedi chiamata):

    {
    ...,
    "parameters": [
        {
            "series": "2b6c9efb4e83f64e757ce862f8b6c2e0",
            "param": "East-West_Displacement",
            "unit": "mm",
            "cmap": [
                [
                    -0.25,
                    "#FFFF00",
                    0
                ],
                [
                    -0.125,
                    "#00FF00",
                    10
                ],
                [
                    0,
                    "#0000FF",
                    20
                ],
                [
                    0.5,
                    "#FF00FF",
                    60
                ],
                [
                    1,
                    "#AEBD00",
                    100
                ]
            ]
        },
        ...
    ],
    ...
}

    dove ogni cmap un array ordinato di triplette utile alla creazione della relativa colorbar. Ogni tipletta fornisce questa infromazione:

    cmap[x][0]: etichetta del valore, possibilmente da piazzare sulla colorbar alla giusta posizione.

    cmap[x][1]: colore in esadecimale.

    cmap[x][2]: posizione dell'ancora del gradiente per il rispettivo colore, nonch per l'eventuale etichetta del valore. La posizione espressa in percentuale sull'intera colorbar.

    29 Marzo 2021

    inserito il concetto di "info" nelle richieste a feature: Con questo parametro si riescono a centralizzare le chiamate che si focalizzano sulle feature e si separano le informazioni di secondo livello ad esse legate. In questo modo la risposta di default risulta piuttosto leggera ma diventa uno strumento potente e versatile specificando esplicitamente l'estrazione di iformazioni accessorie. Attualmente il parametro "info" accetta un array in cui pu essere richiesta l'estrazione di queste info:

    • geometry: il GeoJson della geometria della feature
    • capabilities: le possibili interrogazioni effetuabili sulla feature un array che contiene uno o pi di questi valori: "data","map","cloud". la feature risponder a chiamate del tipo /query/{capability}. Nel caso l'array fosse vuoto la feature non risponderebbe ad alcuna interrogazione.
    • parameters: l'elenco dei parametri associati alla feature come riportati dal'API GET /parametes/. Quest'ultima API diviene inutile e sar eliminata in futuro
    • timeline: la timeline dei dati associati ai parametri della feature. Utile per lo pi per le query map e cloud
    • data: informazioni specifiche di per la capability "data"
    • map: informazioni specifiche per la capability "map" (il formato esattamente quello del nodo "map" del backbone di geoframe)
    • cloud: informazioni specifiche per la capability "cloud"

    • Delineato il modello di scrittura dei Source plugin secondo un template univoco. Ogni plugin potr essere un modulo python definito come segue:

    deve definire tante classi marshmallow.Schema quante sono le azioni che vengono prese in carico dal Template. Marsmallow un serializzatore di oggetti python. Lo schema definito servir per definire i campi in ingresso per ogni azione e fare i check dei valori in ingresso.

    Il nome delle classi Schema deve seguire questa sintassi: class {Action}Schema(marshmallow.Schema) dove {Action} il nome dell'azione (es.: config, feed, ..) con l'iniziale maiuscola. Nella classe vengono definiti i tipi dei campi (marshmallow.fields cfr. https://marshmallow.readthedocs.io/en/stable/ ). ATTENZIONE: in caso fosse necessario l'ingresso di file o comunque oggetti blob dovr essere utilizzato come field la classe hielen2.utils.LocalFile. In questo modo il sistema risolver la chiamata API salvando in locale lo stream dei dati associato a quel determinato field, il quale sar accessibile al template attraverso un path che verr fornito insieme agli altri campi al momento della chiamata del metodo di gestione dell'azione (vedi sotto).

    deve implementare una classe Source(hielen2.datalink.HielenSource) che esponga tanti metodi quante sono le dichiarazioni di Schema seguendo questa sintassi: il metodo di gestione dell'azione deve chiamarsi come l'azione stessa ( tutto in minuscolo ). Le classi estese sfrutteranno il metodo __init__ della superclasse in modo da avere a disposizione tutto quello di cui necessitano.

    Questo modello permette di svincolare i template dalla necessit di conoscere a priori le azioni ammmissibili per il sistema. Infatti, facendo introspezione su un template che segua le regole di sintassi sar sempre possibile conoscere le azioni definite ed esternalizzarle al front-end che in base alle definizioni delle classi di Schema delle azioni, sar sempre in grado di instanziare una form adeguata.
